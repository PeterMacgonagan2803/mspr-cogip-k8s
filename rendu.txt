Résumé exécutif
Ce dossier constitue le livrable final de la MSPR TPRE961, réalisé dans le cadre de la certification professionnelle Expert en Informatique et Système d'Information (RNCP35584 — Bloc 2). Il documente le projet mené pour le compte de la société COGIP, suite à un appel d'offres remporté pour le déploiement d'une infrastructure Kubernetes au service du groupe Tesker.
La solution retenue repose sur la distribution K3s de Kubernetes, déployée sur serveur OVH dédié (Proxmox VE), et orchestrée via la chaîne create-template.sh / Terraform / Ansible. L'ERP Odoo est exposé en HTTPS via l'Ingress Traefik natif K3s avec TLS autosigné et stockage persistant NFS. Les images Docker utilisent le dépôt bitnamilegacy (migration Bitnami).
Les fichiers de configuration complets (Packer alternatif, Terraform, Ansible K3s et Odoo) sont disponibles dans les Annexes A à D et dans le dépôt Git versionné (github.com/PeterMacgonagan2803/mspr-cogip-k8s). Le corps du document présente les choix, la démarche et les résultats.

1. Contexte et présentation du projet
1.1 La société COGIP et son besoin
La société COGIP (COnglomérat Général d'Informatique Professionnelle) est un éditeur de logiciels de gestion d'entreprise. Elle propose l'ERP libre Odoo pour les PME et son ERP propriétaire CogiCloud pour les grandes entreprises. COGIP externalise son infrastructure informatique via des appels d'offres pour rester concentrée sur son cœur de métier : le développement logiciel.
COGIP a récemment décroché un contrat stratégique avec le groupe Tesker, fabricant de véhicules électriques en forte croissance. Ce contrat impose une infrastructure hautement disponible, scalable, et capable de supporter un Plan de Reprise d'Activité (PRA). Notre entreprise a remporté l'appel d'offres correspondant grâce à notre expertise sur la chaîne tooling Kubernetes / Packer / Terraform / Ansible.
1.2 Périmètre du PoC
Le livrable est un Proof of Concept démontrant le déploiement de l'ERP Odoo sur un cluster Kubernetes, entièrement piloté par Infrastructure as Code. Les contraintes du cahier des charges sont :
Cluster Kubernetes : 1 control-plane + 2 workers minimum.
Outils IaC obligatoires : Packer pour les images, Terraform pour le provisionnement, Ansible pour la configuration et le déploiement applicatif.
Versioning Git de l'ensemble des fichiers de configuration.
Odoo accessible via Ingress HTTPS (certificats autosignés acceptés).
Stockage persistant pour Odoo et PostgreSQL.
Solution reproductible constituant la base d'un PRA.

2. Mission 1 — Choix des technologies et justification
2.1 Tableau de synthèse

2.2 Justification de K3s
K3s a été retenu face à RKE2, K0s et MicroK8s pour trois raisons principales. Premièrement, sa légèreté : il ne nécessite que 512 Mo de RAM sur le control-plane, ce qui est bien adapté aux ressources du laboratoire. Deuxièmement, ses composants intégrés (ServiceLB, Traefik, SQLite/etcd) réduisent significativement la complexité de déploiement et le nombre d'étapes Ansible. Troisièmement, son installation via un seul binaire et une commande curl rend le playbook de déploiement simple, idempotent et facilement maintenable. K3s est certifié CNCF et supporte les mêmes manifests Kubernetes qu'une distribution complète.
2.3 Justification de l'approche baremetal Proxmox
Proxmox VE étant disponible sur le laboratoire du campus, ce choix élimine tout coût cloud et tout délai administratif. Le provider Terraform bpg/proxmox permet un provisionnement entièrement déclaratif. Contrairement à une solution cloud managée, cette approche baremetal démontre une maîtrise complète de la chaîne IaC de bout en bout, du template d'image à l'applicatif déployé.

5. Mission 4 — Préparation du template VM (Image Cloud Ubuntu + cloud-init)
5.1 Approche retenue : image cloud officielle + script bash
Plutôt qu'une installation ISO via Packer (plus longue, dépendante d'un serveur HTTP pour l'autoinstall), l'approche retenue utilise directement l'image cloud officielle Ubuntu 22.04 LTS au format qcow2, qui supporte nativement cloud-init. Le script setup/create-template.sh orchestre la création du template Proxmox (ID 9000) en ~3 minutes au lieu de 15–20 minutes avec Packer + ISO.
Les fichiers Packer (répertoire packer/) sont conservés dans le dépôt comme approche alternative documentée, utilisables dans des environnements où l'image cloud officielle n'est pas disponible.
5.2 Ce que crée le script create-template.sh
Le script s'exécute en quatre phases successives sur le serveur Proxmox (en SSH root) :
Téléchargement de l'image cloud officielle Ubuntu 22.04 Jammy (format qcow2, ~700 Mo).
Création de la VM Proxmox avec : 2 vCPU, 2048 Mo RAM, bridge réseau vmbr1, QEMU Guest Agent activé.
Configuration cloud-init par défaut : utilisateur ubuntu, IP DHCP (Terraform injectera l'IP statique au clonage), agent QEMU activé.
Conversion en template Proxmox (qm template 9000) — prêt à être cloné par Terraform via clone { vm_id = 9000 }.

6. Mission 5 — Terraform : provisionnement de l'infrastructure
6.1 Principe et organisation
Terraform est responsable du provisionnement de l'ensemble des 4 VMs sur Proxmox. Toutes les ressources sont déclarées de façon déclarative : Terraform calcule les modifications nécessaires, les présente en dry-run (plan), puis les applique. En cas de reconstruction post-incident, un simple terraform apply suffit à recréer l'infrastructure à l'identique depuis le template cloud-init.
Le projet est organisé en six fichiers : providers.tf (configuration du provider bpg/proxmox), variables.tf (déclaration des paramètres), main.tf (ressources VMs), inventory.tf (génération de l'inventaire Ansible), outputs.tf (sorties), et un fichier template hosts.yml.tftpl. Les valeurs sensibles (mot de passe Proxmox, clé SSH) sont isolées dans terraform.tfvars, exclu du versioning via .gitignore.
6.2 Fonctionnement du main.tf
La ressource centrale est une proxmox_virtual_environment_vm déclarée avec for_each sur une map locale de 4 entrées : k3s-server (VMID 200, IP 10.10.10.10), k3s-worker-1 (201, .11), k3s-worker-2 (202, .12) et nfs-server (203, .13). Chaque VM clone le template 9000 (full clone), reçoit une IP statique via cloud-init, et est étiquetée avec son rôle. Le provider bpg/proxmox a été préféré à telmate/proxmox pour sa meilleure gestion de l'authentification par mot de passe et son support SSH natif.
En sortie, le fichier inventory.tf utilise templatefile() pour générer automatiquement le fichier ansible/inventory/hosts.yml avec les IPs réelles des VMs. Cela garantit la cohérence entre l'état Terraform et l'inventaire Ansible sans intervention manuelle. La version K3s cible est paramétrée via la variable k3s_version (défaut : v1.29.2+k3s1).
6.3 Workflow d'exécution
Le déploiement suit quatre étapes : initialisation du provider (terraform init), validation syntaxique (terraform validate), planification avec aperçu des ressources à créer (terraform plan), puis application effective (terraform apply). La destruction complète de l'infrastructure est possible via terraform destroy, ce qui facilite les cycles de tests.

7. Mission 6 — Ansible : déploiement de Kubernetes K3s
7.1 Organisation des rôles Ansible
Ansible prend en charge la configuration de toutes les VMs à partir de l'inventaire généré par Terraform. L'organisation suit une structure en rôles : common (configuration de base sur toutes les VMs), k3s-server (control-plane), k3s-agent (workers), nfs-server (stockage) et deploy-odoo (applicatif). Les secrets (mots de passe Odoo, PostgreSQL) sont gérés via Ansible Vault (vault.yml).
7.2 Rôle common — Configuration de base
Appliqué à toutes les VMs, ce rôle prépare l'environnement K3s : mise à jour APT, installation des paquets (curl, jq, nfs-common), configuration /etc/hosts pour la résolution inter-nœuds, désactivation du swap (requis par Kubernetes), chargement des modules noyau br_netfilter et overlay, et configuration sysctl (ip_forward, bridge-nf-call-iptables).
7.3 Rôle k3s-server — Control-plane
Installe K3s en mode serveur (version v1.29.2+k3s1) avec les options --write-kubeconfig-mode 644, --tls-san pour l'IP du control-plane, et --cluster-init. Après installation, le rôle attend que K3s soit opérationnel, récupère le node-token pour les workers, sauvegarde le kubeconfig et installe Helm sur le control-plane.
7.4 Rôle k3s-agent — Workers
Installe K3s en mode agent avec l'URL du serveur (https://10.10.10.10:6443) et le token récupéré. Les workers rejoignent le cluster séquentiellement (serial: 1) pour éviter les conflits, et chaque jonction est vérifiée via kubectl get nodes avant de passer au suivant.
7.5 Résultat obtenu
À l'issue du playbook, le cluster est opérationnel avec ses trois nœuds et les composants natifs K3s (CoreDNS, Traefik, ServiceLB, Metrics Server) en état Running :

8. Mission 7 — Ansible : déploiement d'Odoo sur le cluster
8.1 Approche : Ansible + kubernetes.core.helm sans cert-manager
Le déploiement applicatif utilise la collection Ansible kubernetes.core (module helm) pour orchestrer deux charts Helm. Traefik est déjà intégré à K3s, il n'est pas redéployé. Les certificats TLS sont gérés via l'option ingress.selfSigned: true du chart Bitnami Odoo, sans cert-manager (PoC). Les mots de passe applicatifs sont gérés via Ansible Vault (vault.yml).
8.2 Chaîne de déploiement
Le playbook ansible/playbooks/deploy-odoo.yml orchestre trois étapes séquentielles :
NFS Subdir External Provisioner (chart nfs-subdir-external-provisioner) : crée la StorageClass nfs-client pointant vers le serveur NFS (10.10.10.13). Le pod provisioner peut être placé sur un nœud spécifique via nodeSelector.
Chart Bitnami Odoo (dépôt https://charts.bitnami.com/bitnami, namespace odoo) : déploie Odoo avec PostgreSQL intégré (sous-chart Bitnami), persistance sur nfs-client, service type ClusterIP (pas LoadBalancer pour éviter le statut <pending> sans cloud-controller), TLS autosigné via ingress.selfSigned: true, accès HTTPS via Traefik natif K3s. Les images conteneurs utilisent le dépôt bitnamilegacy (docker.io/bitnamilegacy/*) avec global.security.allowInsecureImages: true suite à la migration des tags Bitnami. Timeout Helm porté à 20 minutes pour le premier pull d'images sur cluster TP.
Ingress Traefik : l'annotation traefik.ingress.kubernetes.io/router.entrypoints: web,websecure garantit l'accès aussi bien en HTTP qu'en HTTPS. L'URL finale est https://odoo.local (certificat autosigné — avertissement navigateur attendu en PoC).
8.3 Validation du déploiement
À l'issue du playbook, tous les pods sont en état Running, les PVCs sont liés aux volumes NFS, et Odoo est accessible depuis https://odoo.local. Le playbook destroy.yml permet la suppression propre des releases Helm et des namespaces.

A.3 scripts/base.sh
#!/bin/bash
set -euxo pipefail

apt-get update && apt-get upgrade -y
apt-get install -y curl wget git vim python3 python3-pip \
  apt-transport-https ca-certificates gnupg lsb-release \
  qemu-guest-agent open-iscsi nfs-common

systemctl enable qemu-guest-agent

# Désactivation du swap
swapoff -a
sed -i "/\sswap\s/d" /etc/fstab

# Modules noyau requis par Kubernetes
cat <<EOF | tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF
modprobe overlay && modprobe br_netfilter

# Paramètres sysctl
cat <<EOF | tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-ip6tables = 1
net.bridge.bridge-nf-call-iptables  = 1
net.ipv4.ip_forward                 = 1
EOF
sysctl --system

# Nettoyage avant snapshot
apt-get autoremove -y && apt-get clean
rm -f /etc/ssh/ssh_host_*
truncate -s 0 /etc/machine-id

locals {
  vms = {
    "k3s-cp-01"     = { vm_id=101, ip="192.168.1.101", cores=2, memory=4096, disk_gb=20, role="control_plane" }
    "k3s-worker-01" = { vm_id=102, ip="192.168.1.102", cores=2, memory=4096, disk_gb=30, role="worker" }
    "k3s-worker-02" = { vm_id=103, ip="192.168.1.103", cores=2, memory=4096, disk_gb=30, role="worker" }

resource "proxmox_virtual_environment_vm" "nodes" {
  for_each  = local.vms
  name      = each.key
  vm_id     = each.value.vm_id
  node_name = var.proxmox_node
  tags      = ["k3s", each.value.role, "terraform"]

  clone { vm_id = var.template_vm_id; full = true }

  cpu    { cores = each.value.cores; type = "x86-64-v2-AES" }
  memory { dedicated = each.value.memory }

  disk {
    datastore_id = "local-lvm"
    interface    = "virtio0"
    size         = each.value.disk_gb
  }

  network_device { bridge = "vmbr0"; model = "virtio" }

  initialization {
    ip_config {
      ipv4 { address = "${each.value.ip}/24"; gateway = var.network_gateway }
    }
    user_account { username = "ubuntu"; keys = [var.ssh_public_key] }
    dns { servers = [var.dns_server] }
  }

  operating_system { type = "l26" }
  agent   { enabled = true }
  on_boot = true
}

C.1 01-nfs.yml
---
- name: Configuration du serveur NFS
  hosts: nfs
  become: true
  tasks:

    - name: Installation de nfs-kernel-server
      ansible.builtin.apt:
        name: nfs-kernel-server
        state: present
        update_cache: true

    - name: Création du répertoire de partage NFS
      ansible.builtin.file:
        path: "{{ nfs_share_path }}"
        state: directory
        owner: nobody
        group: nogroup
        mode: "0777"

    - name: Configuration de l'export NFS
      ansible.builtin.lineinfile:
        path: /etc/exports
        line: "{{ nfs_share_path }} 192.168.1.0/24(rw,sync,no_subtree_check,no_root_squash)"
        create: true

    - name: Application des exports
      ansible.builtin.command: exportfs -ra

    - name: Démarrage et activation de nfs-kernel-server
      ansible.builtin.systemd:
        name: nfs-kernel-server
        state: started
        enabled: true
C.2 02-k3s-install.yml
---
- name: Installation K3s — control-plane
  hosts: control_plane
  become: true
  tasks:

    - name: Installation de K3s
      ansible.builtin.shell: |
        curl -sfL https://get.k3s.io | \
          INSTALL_K3S_VERSION="{{ k3s_version }}" \
          K3S_TOKEN="{{ lookup('password', '/tmp/k3s_token chars=ascii_letters,digits length=64') }}" \
          sh -s - server --write-kubeconfig-mode 644
      args:
        creates: /usr/local/bin/k3s

    - name: Attente du node-token
      ansible.builtin.wait_for:
        path: /var/lib/rancher/k3s/server/node-token
        state: present
        timeout: 120

    - name: Lecture du token K3s
      ansible.builtin.slurp:
        src: "{{ k3s_token_file }}"
      register: k3s_token_b64

    - name: Enregistrement du token en fact
      ansible.builtin.set_fact:
        k3s_token: "{{ k3s_token_b64.content | b64decode | trim }}"

    - name: Récupération du kubeconfig
      ansible.builtin.fetch:
        src:  /etc/rancher/k3s/k3s.yaml
        dest: "~/.kube/config"
        flat: true

- name: Jonction des workers
  hosts: workers
  become: true
  tasks:

    - name: Installation K3s agent
      ansible.builtin.shell: |
        curl -sfL https://get.k3s.io | \
          INSTALL_K3S_VERSION="{{ k3s_version }}" \
          K3S_URL="https://{{ k3s_server_ip }}:6443" \
          K3S_TOKEN="{{ hostvars[groups['control_plane'][0]]['k3s_token'] }}" \
          sh -s - agent
      args:
        creates: /usr/local/bin/k3s-agent

    - name: Activation du service K3s agent
      ansible.builtin.systemd:
        name: k3s-agent
        state: started
        enabled: true

- name: Validation du cluster
  hosts: control_plane
  become: true
  tasks:

    - name: Vérification — tous les nœuds Ready
      ansible.builtin.command: kubectl get nodes --no-headers
      register: nodes_status
      until: nodes_status.stdout_lines | length == 3
             and nodes_status.stdout is not search("NotReady")
      retries: 20
      delay: 15

    - name: Affichage de l'état du cluster
      ansible.builtin.debug:
        msg: "{{ nodes_status.stdout_lines }}"

Annexe D — Playbook Ansible Odoo complet (03-odoo.yml)
---
- name: Déploiement Odoo sur le cluster K3s
  hosts: control_plane
  become: true
  collections:
    - kubernetes.core
  tasks:

  # ── 1. Helm ──────────────────────────────────────────────────────
    - name: Téléchargement du script Helm
      ansible.builtin.get_url:
        url:  https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3
        dest: /tmp/get-helm.sh
        mode: "0755"

    - name: Installation de Helm
      ansible.builtin.shell: /tmp/get-helm.sh
      args:
        creates: /usr/local/bin/helm

  # ── 2. NFS StorageClass ──────────────────────────────────────────
    - name: Ajout dépôt nfs-subdir-external-provisioner
      kubernetes.core.helm_repository:
        name:     nfs-subdir-external-provisioner
        repo_url: https://kubernetes-sigs.github.io/nfs-subdir-external-provisioner/

    - name: Déploiement NFS provisionner
      kubernetes.core.helm:
        name:              nfs-subdir-external-provisioner
        chart_ref:         nfs-subdir-external-provisioner/nfs-subdir-external-provisioner
        release_namespace: kube-system
        kubeconfig:        /etc/rancher/k3s/k3s.yaml
        values:
          nfs:
            server: "{{ nfs_server_ip }}"
            path:   "{{ nfs_share_path }}"
          storageClass:
            name:          nfs-client
            defaultClass:  true
            reclaimPolicy: Retain

  # ── 3. cert-manager ──────────────────────────────────────────────
    - name: Ajout dépôt jetstack
      kubernetes.core.helm_repository:
        name: jetstack
        repo_url: https://charts.jetstack.io

    - name: Déploiement cert-manager
      kubernetes.core.helm:
        name:              cert-manager
        chart_ref:         jetstack/cert-manager
        chart_version:     "v1.14.4"
        release_namespace: cert-manager
        create_namespace:  true
        kubeconfig:        /etc/rancher/k3s/k3s.yaml
        values:
          installCRDs: true

    - name: ClusterIssuer autosigné
      kubernetes.core.k8s:
        kubeconfig: /etc/rancher/k3s/k3s.yaml
        state: present
        definition:
          apiVersion: cert-manager.io/v1
          kind: ClusterIssuer
          metadata:
            name: selfsigned-issuer
          spec:
            selfSigned: {}

  # ── 4. Traefik ───────────────────────────────────────────────────
    - name: Ajout dépôt Traefik
      kubernetes.core.helm_repository:
        name: traefik
        repo_url: https://helm.traefik.io/traefik

    - name: Déploiement Traefik
      kubernetes.core.helm:
        name:              traefik
        chart_ref:         traefik/traefik
        release_namespace: traefik
        create_namespace:  true
        kubeconfig:        /etc/rancher/k3s/k3s.yaml
        values:
          ports:
            web:
              redirectTo: websecure
            websecure:
              tls:
                enabled: true
          service:
            type: LoadBalancer

  # ── 5. Namespace + Odoo ──────────────────────────────────────────
    - name: Namespace odoo
      kubernetes.core.k8s:
        kubeconfig: /etc/rancher/k3s/k3s.yaml
        state: present
        definition:
          apiVersion: v1
          kind: Namespace
          metadata:
            name: odoo

    - name: Ajout dépôt Bitnami
      kubernetes.core.helm_repository:
        name: bitnami
        repo_url: https://charts.bitnami.com/bitnami

    - name: Déploiement Odoo
      kubernetes.core.helm:
        name:              odoo
        chart_ref:         bitnami/odoo
        chart_version:     "26.0.0"
        release_namespace: odoo
        kubeconfig:        /etc/rancher/k3s/k3s.yaml
        wait:              true
        timeout:           "600s"
        values:
          odooEmail:    admin@cogip.local
          odooPassword: "CogipSecure2024!"
          service:
            type: ClusterIP
          persistence:
            enabled:      true
            storageClass: nfs-client
            size:         8Gi
          postgresql:
            enabled: true
            primary:
              persistence:
                enabled:      true
                storageClass: nfs-client
                size:         4Gi
          ingress:
            enabled: false

  # ── 6. Certificat TLS + IngressRoute ─────────────────────────────
    - name: Certificat TLS autosigné pour Odoo
      kubernetes.core.k8s:
        kubeconfig: /etc/rancher/k3s/k3s.yaml
        state: present
        definition:
          apiVersion: cert-manager.io/v1
          kind: Certificate
          metadata:
            name:      odoo-tls
            namespace: odoo
          spec:
            secretName: odoo-tls-secret
            issuerRef:
              name: selfsigned-issuer
              kind: ClusterIssuer
            dnsNames:
              - "{{ odoo_domain }}"

    - name: IngressRoute Traefik pour Odoo
      kubernetes.core.k8s:
        kubeconfig: /etc/rancher/k3s/k3s.yaml
        state: present
        definition:
          apiVersion: traefik.containo.us/v1alpha1
          kind: IngressRoute
          metadata:
            name:      odoo-ingressroute
            namespace: odoo
          spec:
            entryPoints: [websecure]
            routes:
              - match: Host(`{{ odoo_domain }}`)
                kind: Rule
                services:
                  - name: odoo
                    port: 8069
            tls:
              secretName: odoo-tls-secret
